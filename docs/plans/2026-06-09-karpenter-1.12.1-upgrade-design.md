# Design: Karpenter 1.12.1 Upgrade + Cluster Setup

**Date:** 2026-06-09
**Author:** Viktor Vedmich (with Claude Code)
**Status:** Approved

## Goal

Bring the repository up to the current Karpenter release (1.12.1), then
deterministically create both demo clusters (`k-basic`, `k-hl`) in `eu-north-1`
for an upcoming Karpenter workshop. Kubernetes stays at 1.35.

## Context (verified, not from memory)

- Latest Karpenter release: **v1.12.1** (1.10/1.11/1.12 released after the repo's
  1.9.0 fallback). Verified via GitHub releases API on 2026-06-09.
- CRD APIs `karpenter.sh/v1` and `karpenter.k8s.aws/v1` are stable since 1.0.
  Release notes for 1.10–1.12 contain only features and bug fixes — **no breaking
  CRD changes**. Existing NodePool / EC2NodeClass manifests need no schema edits.
- 1.11.0 carried a CPU-usage regression (upstream issue kubernetes-sigs/karpenter#2954);
  **1.12.1 does not** — so 1.12.1 is the safe target.
- CloudFormation template URL embedded in `create-cluster.sh`
  (`.../v${KARPENTER_VERSION}/website/content/en/preview/.../cloudformation.yaml`)
  returns **HTTP 200** for `v1.12.1`. The `/preview/` path is correct; a `/v1.12/`
  path 404s.
- Helm chart `oci://public.ecr.aws/karpenter/karpenter --version 1.12.1` is
  available (appVersion 1.12.1).
- AWS account `972819599786` (Isengard Admin). No EKS clusters in eu-north-1 or
  us-east-1; no Karpenter CloudFormation stacks. Clean slate.
- `eu-north-1`: 3 AZs (a/b/c), `c5.2xlarge` offered in all three, Spot vCPU quota
  = 10000 — ample headroom for the 3000-pod highload test.

## Known environment risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| Shell region (`us-east-1`) ≠ config (`eu-north-1`) | High | Run every command with explicit `AWS_DEFAULT_REGION=eu-north-1` |
| `aws-cli v1.40` installed (README expects v2) | Medium | v1 covers eks/cloudformation/ssm/ec2 calls used here; if `update-kubeconfig`/`describe-images` fail, flag and recommend v2 |
| `helm 4.2` installed (Karpenter tested on 3.x) | Low-Med | OCI `helm upgrade --install` works on helm 4; fall back to helm 3 if chart install fails |
| Spot capacity / ICE during 3000-pod test | Low | highload NodePool already diversified across c5/c6i/c7i/m5/m6i/m7i/r5/r6i/r7i × 3 AZ; Karpenter retries other types on ICE |
| Failure on 2nd cluster | Medium | Step-by-step run (Approach B) — basic stays intact, resume highload independently |

## Code changes (before any cluster is created)

| File | Change | Reason |
|------|--------|--------|
| `config.env` | `KARPENTER_VERSION_FALLBACK`: `1.9.0` → `1.12.1` | Fallback should reflect target version |
| `manifests/highload/ec2nodeclass.yaml.tpl` | `maxPods: 150` → `200` | Resolve mismatch with README/CLAUDE.md (both claim 200) |
| `README.md` | fallback `1.9.0`→`1.12.1`; version examples → `1.12.1`; sample dates `26-03-10` → `26-06-09` | Keep docs in sync |

**Explicitly NOT changed:** CRD manifests (stable API), script logic (URLs/chart
verified working), K8s version (stays 1.35), config.env region default (stays
`eu-north-1`, overridden per-command).

## Execution approach: B (step-by-step, orchestrated by Claude)

Rejected Approach A (`setup-all.sh` one-shot): atomic but hard to resume mid-run
and contains an interactive `read -p`. Approach B mirrors the `/setup-demo`
command path, gives per-step visibility and failure recovery.

Steps:

1. Apply the three code changes; commit.
2. Preflight: AWS identity / region / no conflicting clusters / tool versions
   (largely already verified).
3. Create clusters sequentially (eksctl does not parallelize well; ~15–20 min each):
   - `create-cluster.sh basic k-basic-26-06-09` — CloudFormation IAM + EKS
   - `create-cluster.sh highload k-hl-26-06-09` — same
   Long-running steps run via background Bash with periodic status checks.
4. Install Karpenter 1.12.1 on each (Helm + NodePool/EC2NodeClass manifests).
5. Deploy monitoring (metrics-server + kube-ops-view) on both.
6. Rename kubectl contexts to `k-basic` / `k-hl`.

All cluster-touching commands carry `AWS_DEFAULT_REGION=eu-north-1`.

## Success criteria

- `kubectl get nodes` on both contexts shows managed nodes (tainted
  `CriticalAddonsOnly`).
- Karpenter pods `Running` in `kube-system`, image tag 1.12.1.
- `kubectl get nodepools` and `kubectl get ec2nodeclasses` report Ready.
- kube-ops-view reachable on both clusters.

## Rollback

`./scripts/teardown.sh k-basic-26-06-09 k-hl-26-06-09` removes EKS + CloudFormation
+ IAM. Clusters are independent; a partial result does not block cleanup.
